# Spec: GET /templates/{slug}/spec — Requirements

**Spec ID**: api-spec-endpoint
**Date**: 2026-05-12
**Author**: Orlando Bruno
**Status**: Draft
**Feature**: FR-18 — Template Spec Endpoint

---

## 1. PRD References

PRD references are all located at `_Design/01_PRD/prd.md`.

- FR-17: Every template pack shall include a `spec.yaml` — a machine-readable document describing all fields, types, constraints, and layout context — to enable LLM-assisted profile generation.
- FR-18: The HTTP API shall expose a `GET /templates/{slug}/spec` endpoint that returns the full `spec.yaml` content for a given template, so remote clients can fetch it without filesystem access before generating a profile.
- FR-10: The system shall provide an HTTP API (`GET /`, `GET /templates`, `GET /templates/{slug}/spec`, `POST /render`, ...) as an optional `api` extra. (FR-10 explicitly lists this endpoint — it is missing from the current implementation.)
- Story 5: LLM-Assisted Profile Generation — an LLM or automation client fetches a template's `spec.yaml` via the API to generate a correctly structured profile YAML before calling `/render`.
- Goal 6: Enable LLM-assisted profile generation — each template exposes a machine-readable `spec.yaml` so an LLM can generate a correctly structured, constraint-aware profile adapted to any chosen design.
- NFR-09: The HTTP API shall not expose internal filesystem paths in error responses.

---

## 2. Overview

The `GET /templates/{slug}/spec` endpoint closes the last gap in the LLM-assisted profile generation workflow. Currently, the API can list available templates (`GET /templates`) and render a profile to PDF (`POST /render`), but it provides no way for a remote client to fetch the template's field schema before authoring a profile.

The `spec.yaml` file in each template directory is the machine-readable contract: it documents every field, its type, constraints (`max_chars`, `max_items`), and layout context. Without API access to it, LLMs or automation clients must have filesystem access to the running container — which defeats the purpose of exposing an HTTP API.

This endpoint returns the raw `spec.yaml` text so any HTTP client (LLM, automation tool, developer) can retrieve the schema, generate a valid profile, and immediately call `POST /render` — all without filesystem access.

## 3. User Stories

### Story 5A: LLM Fetches Spec Before Profile Generation

> As an LLM or automation agent calling the Paperwork API,
> I want to fetch the raw `spec.yaml` for a template via `GET /templates/{slug}/spec`,
> so that I can generate a correctly structured, constraint-aware profile YAML without filesystem access or manual schema documentation.

Acceptance criteria:
- [ ] `GET /templates/classic/spec` returns `200 OK` with the full text of `templates/classic/spec.yaml`.
- [ ] The `Content-Type` header is `text/plain; charset=utf-8`.
- [ ] The response body is the exact raw YAML text — no JSON wrapping, no parsing, no transformation.
- [ ] A profile generated by an LLM using the returned spec passes `paperwork validate` against the same template without manual correction.

### Story 5B: Remote Client Discovers Available Specs

> As a remote API client building a CV generation pipeline,
> I want `GET /templates` to indicate which templates have a spec file,
> so that I know which templates support the full LLM-assisted workflow before fetching a spec.

Acceptance criteria:
- [ ] `GET /templates` continues to return the full `TemplateMeta` JSON for each template.
- [ ] The `spec_file` field is present in each template's metadata (null when absent).
- [ ] No regression in `GET /templates` behavior after adding the new route.

### Story 5C: Developer Inspects Template Schema via curl

> As a developer integrating with the API,
> I want to run `curl http://localhost:8000/templates/classic/spec` and see the spec.yaml content printed in my terminal,
> so that I can understand the template's data contract without opening the container filesystem.

Acceptance criteria:
- [ ] `curl http://localhost:8000/templates/classic/spec` outputs the spec.yaml text to stdout.
- [ ] `curl -i http://localhost:8000/templates/classic/spec` shows `Content-Type: text/plain; charset=utf-8`.

## 4. Functional Requirements

| ID | Requirement | Priority |
|----|-------------|----------|
| FR-18.1 | The API shall expose a `GET /templates/{slug}/spec` route. | High |
| FR-18.2 | The endpoint shall return the raw text content of the template's `spec_file` (as declared in `template.yaml`) with `Content-Type: text/plain; charset=utf-8`. | High |
| FR-18.3 | The endpoint shall return `HTTP 404` with a structured JSON error body when `slug` does not resolve to a registered template. The error message shall list available slugs. | High |
| FR-18.4 | The endpoint shall return `HTTP 404` with a structured JSON error body when the resolved template has no `spec_file` declared (field is null or absent in `template.yaml`). | High |
| FR-18.5 | The endpoint shall return `HTTP 500` with a structured JSON error body when `spec_file` is declared but the file does not exist on disk. This indicates a packaging/deployment error. | High |
| FR-18.6 | Error messages shall not expose absolute filesystem paths. | High |
| FR-18.7 | The route shall be registered inside `create_router()` in `src/paperwork/api/routes.py` using the existing `TemplateRegistry` accessed through the `engine` closure. | Medium |

## 5. Non-Functional Requirements

| ID | Category | Requirement |
|----|----------|-------------|
| NFR-18.1 | Performance | The endpoint shall respond in under 500 ms for any spec file up to 50 KB (local disk read, no computation). |
| NFR-18.2 | Correctness | The `Content-Type` response header shall be `text/plain; charset=utf-8`. The body shall be the exact bytes of the spec file — no transformation, no JSON encoding. |
| NFR-18.3 | Security | No absolute filesystem paths shall appear in `4xx` or `5xx` error response bodies (NFR-09 compliance). |
| NFR-18.4 | Testability | The route shall be covered by unit tests using FastAPI `TestClient` for all four response cases (200, 404-no-slug, 404-no-spec, 500-missing-file). |
| NFR-18.5 | Maintainability | No new dependencies are introduced; the implementation uses only FastAPI `Response`, `HTTPException`, and Python `pathlib.Path`. |

## 6. Success Criteria

- [ ] `GET /templates/classic/spec` returns `200 OK` with the exact text of `templates/classic/spec.yaml`.
- [ ] `GET /templates/unknown-slug/spec` returns `404` with a JSON body naming available templates.
- [ ] `GET /templates/{slug}/spec` where slug has no `spec_file` returns `404` with message "Template '{slug}' has no spec file defined".
- [ ] `GET /templates/{slug}/spec` where `spec_file` is declared but missing from disk returns `500` (not a crash).
- [ ] No absolute filesystem path appears in any error response body.
- [ ] `GET /templates` continues to work after the new route is added (no regression).
- [ ] Four tests pass: `test_get_spec_200`, `test_get_spec_404_unknown_slug`, `test_get_spec_404_no_spec_file`, `test_get_spec_500_missing_file`.

---

## 7. Out of Scope

- Uploading or modifying a `spec.yaml` via the API — the API is read-only for template assets.
- Validating that a spec.yaml conforms to any schema — the endpoint returns raw text; validation is the caller's responsibility.
- Serving other template assets (HTML, CSS, SCSS) via the API — only the spec file declared in `spec_file` is exposed.
- Parsing the YAML or transforming it to JSON before returning — raw text only.
- Caching the spec file in memory — file read on every request is sufficient for v0.x usage patterns.
- Authentication or authorization for this endpoint — consistent with the existing API (NFR from PRD: no auth layer).

## 8. Dependencies

- **TemplateRegistry** (`src/paperwork/templates/registry.py`): provides `get_template(slug)` and `get_template_dir(slug)`. Both methods already exist. `get_template()` raises `TemplateNotFoundError` on unknown slug.
- **TemplateMeta** (`src/paperwork/models/template_meta.py`): the `spec_file: Optional[str]` field is already defined. No model changes required.
- **FR-17**: Every built-in template must have a `spec.yaml`. The `classic` template already satisfies this (`spec_file: "spec.yaml"` in `template.yaml`). Additional templates are a prerequisite for full FR-17 compliance but are not a blocker for this endpoint.
- **`spec.yaml` on disk**: The `classic` template's `spec.yaml` exists at `templates/classic/spec.yaml`. The endpoint reads from `RENDERCV_TEMPLATES_DIR` (default: `/app/templates` in Docker, `./templates/` locally).
